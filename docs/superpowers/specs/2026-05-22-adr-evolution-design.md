# `dev-flow`: ADR Evolution — bd as Source of Truth + Templating + Lifecycle

**Date:** 2026-05-22
**Status:** Proposed
**Deciders:** Sean Brandt (`@seanb4t`)
**Supersedes:** —
**Bead:** [fhsk-4xf](../../../README.md) — _ADR evolution skill: update/edit/evolve guidance (scope-expanded)_

## Overview

Shift dev-flow's ADR machinery so that **bd is the source of truth** for ADR content; the markdown files under `docs/adr/` are derived views regenerated from bd state. Add a versioned bd formula for ADR scaffolding, a render script that emits markdown from bd, an `evolve-adr` skill + `/adr` slash command for lifecycle operations (propose / update / supersede / addendum / accept / deprecate / render / migrate), and adr-doctor lint extensions that catch drift between bd state and the committed markdown.

The current state (after PRs #78 and #79) is bd-aware but markdown-canonical: `capture-adrs` writes the body to BOTH bd's description AND the .md file, then they drift independently. This spec inverts that: the bd `decision` bead's `--description` is canonical; the markdown is regenerated; drift is a lint failure.

## Goals

- Make bd the source of truth for ADR content (description body + status + dates + supersession edges).
- Provide a versioned formula (`formula-adr.formula.toml`) that pours decision beads with the canonical body shape using bd's built-in `decision`-type `--validate` checks.
- Provide forward-looking ADR support: `Proposed` (open) → `Accepted` (closed) → optional `Superseded` / `Deprecated`.
- Provide repo-portable migration (`/adr migrate`) so any dev-flow consumer (this repo, holomush, etc.) can backfill legacy ADRs to the new envelope without re-creating bd records.
- Eliminate bd ↔ markdown drift via adr-doctor lint checks.

## Non-goals

- Replacing bd's built-in `decision` type with a custom type. Custom types cannot declare section schemas (verified via deepwiki); the built-in `decision` type's `--validate` enforces `## Decision` / `## Rationale` / `## Alternatives Considered`, which we keep using.
- Storing per-section content in separate bd metadata fields. The body stays in `--description` as canonical markdown sections; only minimal envelope (`adr_deciders`) goes in metadata. Fragmenting prose into metadata keys is anti-idiomatic per the bd design.
- Auto-regenerating markdown on every bd write via hooks. We chose explicit render + adr-doctor drift detection (the user-approved "hybrid" mechanism); zero-touch hook-based sync is rejected as too easy to misfire on note appends.
- Build-time-only markdown (CI-generated, not in git). The markdown stays in git for PR visibility; it's just a derived view.

## Architecture

### bd as source of truth — what lives where

| ADR concept | bd-native representation | Notes |
|-------------|--------------------------|-------|
| Title | bd bead title | Already canonical |
| Body (Context / Decision / Rationale / Alternatives / Consequences) | `--description` (markdown sections within) | bd's `--validate` enforces 3 sections (Decision / Rationale / Alternatives Considered); adr-doctor enforces Context + Consequences as project convention |
| Status (computed at render time) | bd lifecycle + dep edges + labels | See "Computed status" below |
| Date | bd `closed_at` (decision made) OR `created_at` (still Proposed) | No separate `adr_date` field |
| Deciders | Metadata `adr_deciders` | One custom key; comma-separated names. bd's Owner/Assignee mean work-assignment, different semantic. |
| Supersession edges | `bd dep add <new> <old> --type=supersedes` | Existing bd dep type; existing INV-A13 check covers this |
| Addenda | `bd note <id> "addendum: <text>"` | Renderer surfaces as `## Addenda` |
| Markdown file | `docs/adr/<bd-id>-<slug>.md` | Derived view; regenerated by `render-adr` |

**Computed status (pseudocode in the renderer):**

```text
if bd.status == "open":
    return "Proposed"
if bd has incoming supersedes edge (bd dep list --direction=up --type=supersedes):
    return "Superseded by <new-id>"
if bd.labels includes "adr:rejected":
    return "Rejected"
if bd.labels includes "adr:deprecated":
    return "Deprecated"
# closed bead with no special marker:
return "Accepted"
```

**Rejection path:** a Proposed ADR that gets rejected (the team decided against it) is closed with `bd update <id> --add-label adr:rejected` + `bd close <id> --reason="Rejected: <why>"`. Renders as `Status: Rejected`. This is distinct from Deprecation (Accepted-then-retired) and Supersession (replaced-by-another-ADR). A symmetric `/adr reject <id>` command MAY be added in a follow-up; for now operators run the bd commands directly.

### `/adr` slash command surface

| Command | Action |
|---------|--------|
| `/adr init` | One-shot per-repo bootstrap: copy `formula-adr.formula.toml` into `.beads/formulas/`; register the `adr` alias (bd's built-in alias for `decision` may not need this — verify in plan stage); set the `adr:deprecated` label intent. Idempotent. |
| `/adr new` | Retrospective documentation flow (same shape as `capture-adrs` filing): create ADR + auto-close (Accepted). Calls `bd mol pour formula-adr` + `bd close` + `render-adr`. |
| `/adr propose` | Forward-looking flow: create ADR; leave open → renders as `Status: Proposed`. Operator can `/adr accept <id>` later. |
| `/adr update <id>` | Interactive: prompt for which section + new prose → re-compose the full description body and pipe via `bd update <id> --body-file -` (bd's piped-body flag; `--stdin` is an alias) → re-render. Multi-line prose is shell-escape-safe via this flag. |
| `/adr supersede <old-id>` | New-ADR flow (interactive) → `bd dep add <new> <old> --type=supersedes` → ensure old `bd.status=closed` → re-render BOTH markdown files. |
| `/adr addendum <id> --text "..."` | `bd note <id> "addendum: <text>"` → re-render. |
| `/adr accept <id>` | `bd close <id> --reason="Accepted"` → re-render (status flips Proposed → Accepted). |
| `/adr deprecate <id> --reason "..."` | `bd update <id> --add-label adr:deprecated` (preserves existing labels; bd's `--add-label` is the additive flag, distinct from `--set-labels` which replaces) → re-render (status flips Accepted → Deprecated). |
| `/adr render <id>` | Re-render only; no bd mutation. For fixing detected drift. |
| `/adr migrate [--apply]` | Repo-portable backfill (see "Migration" below). Default is dry-run preview; `--apply` mutates. |

### Distinction: `capture-adrs` vs. `/adr`

Both create ADRs. The split:

- **`capture-adrs`** (existing skill, slightly modified): retrospective documentation triggered after `writing-plans` / via the nudge-adr-capture hook. Decision was made in a brainstorming session and is already final. Creates Accepted ADRs (closes the bead immediately). Output: one or more ADRs per spec/plan, all triaged through the user's review loop, then poured via formula.

- **`/adr propose`** (new): forward-looking. Decision is being discussed, not yet final. Creates `Proposed` ADR (bd status=open). Operator runs `/adr accept <id>` later when the team commits, `/adr update <id>` to refine, or rejects via `bd update <id> --add-label adr:rejected` + `bd close <id> --reason="Rejected: <why>"` (renders as `Status: Rejected`; see "Computed status" — Rejected, Deprecated, and Superseded are three distinct terminal states).

Both paths use the same formula and renderer.

### adr-doctor extensions

Five new checks added to `dev-flow/scripts/adr-doctor.sh`:

| Check | INV ref | Logic |
|-------|---------|-------|
| `description_has_context_section` | INV-A20 | For each decision bead with non-empty description, the description contains `## Context` heading |
| `description_has_consequences_section` | INV-A21 | Same, `## Consequences` |
| `markdown_matches_render` | INV-A22 | For each `<bd-id>-*.md`, run `render-adr <bd-id>` to a temp file; hash-compare against the committed .md. Drift = fail with hint "run `/adr render <bd-id>`" or "run `/adr update <id>`". |
| `status_label_coherent` | INV-A23 | If bead has `adr:deprecated` label, bd status MUST be `closed` (deprecated implies decision was once accepted). Catches "labeled deprecated but still open." |
| `adr_deciders_present` | INV-A24 | metadata `adr_deciders` non-empty for any decision bead with bd status `closed` (Accepted/Superseded/Deprecated all require attribution). Warning-only for legacy beads pre-migration; error after migration. |

(The existing INV-A1 through INV-A19 are used across `dev-flow/scripts/adr-doctor.sh` and `dev-flow/skills/capture-adrs/SKILL.md`. Notably INV-A15 is used in adr-doctor.sh and INV-A19 is the existing AskUserQuestion main-session-context constraint in capture-adrs. This spec assigns INV-A20 through INV-A24 to the new checks to avoid collisions; gaps A6/A7/A8/A9/A11/A16/A17/A18 are reserved for future use.)

The existing checks (INV-A4 / A5 / A12 / A13 / A14) are unchanged. The greedy-regex fix from PR #78 stays in place.

### `formula-adr.formula.toml` (structural shape; exact bytes are implementation)

```toml
formula = "formula-adr"
version = 1
description = """
ADR (Architecture Decision Record). Built on bd's built-in `decision` type;
bd's --validate enforces ## Decision / ## Rationale / ## Alternatives Considered.
Project convention additionally requires ## Context and ## Consequences,
enforced by adr-doctor at lint time.

Body authored at pour time via vars. Edit later via /adr update <id>;
render to markdown via /adr render <id>.
"""

[vars.title]
description = "ADR title (imperative — what the decision is)"
required = true

[vars.context]
description = "## Context section prose"
required = true

[vars.decision]
description = "## Decision section prose"
required = true

[vars.rationale]
description = "## Rationale section prose"
required = true

[vars.alternatives]
description = "## Alternatives Considered section prose"
required = true

[vars.consequences]
description = "## Consequences section prose"
required = true

[vars.deciders]
description = "Comma-separated decider names (stored as adr_deciders metadata)"
required = true

[[steps]]
id = "adr-root"
type = "decision"
title = "{{title}}"
description = """## Context

{{context}}

## Decision

{{decision}}

## Rationale

{{rationale}}

## Alternatives Considered

{{alternatives}}

## Consequences

{{consequences}}
"""
labels = ["phase:design"]
metadata = { adr_deciders = "{{deciders}}" }
```

### `render-adr` script behavior

```text
INPUT:   bd-id (positional)
SIDE EFFECT: writes docs/adr/<bd-id>-<slug>.md
EXIT:    0 on success; non-zero on bd query failure, missing required sections, or other unrecoverable errors.

Steps:
  1. bd show <id> --json → extract title, description, metadata, notes, deps, created_at, closed_at, status.
  2. Compute slug: kebab-case title; drop stop-words (a/an/the/for/of/to/in/on/with); cap 60 chars.
  3. Compute status per the pseudocode above (open/supersedes-up/labels/closed).
  4. Compute date: closed_at if present, else created_at; format YYYY-MM-DD.
  5. Read .notes[] for entries starting with "addendum: "; collect into composed "## Addenda" section.
  6. Emit canonical markdown:
       <!-- markdownlint-disable MD013 -->
       <!-- adr-render: source=bd:<id>; do not edit manually; use `/adr update <id>` -->

       # <title>

       **Date:** <date>
       **Status:** <computed-status>
       **Decision:** <id>
       **Deciders:** <metadata.adr_deciders>

       <description body — already contains ## sections>

       [if addenda: ## Addenda heading + entries]

       ## References

       [if outgoing supersedes deps: "Supersedes: <old-id>" with link]
       [if incoming supersedes deps: "Superseded by: <new-id>" with link]
```

The `<!-- adr-render: ... -->` marker on line 2 of every rendered file:

- Signals to readers that the file is generated.
- Prevents human edits silently being lost on next render (operators should use `/adr update`).
- Distinct from the existing `<!-- adr-capture: sha256=... -->` marker that `capture-adrs` writes on the spec; render-adr's marker is on the ADR file itself.

### Migration: `/adr migrate`

Repo-portable. Default is dry-run preview; `--apply` mutates.

```text
For each decision bead in bd (`bd list --type=decision --json`):
  IF metadata.adr_deciders is set:
    → already migrated; skip (idempotent)
  ELSE (legacy bead — created via pre-formula direct capture-adrs path):
    Search docs/adr/ for a file whose basename starts with "<bd-id>-":
      IF file found:
        - Parse `**Deciders:**` line from the file's header
        - DRY-RUN: report "would set adr_deciders=<extracted> on <bd-id>; would re-render"
        - APPLY: bd update <bd-id> --set-metadata adr_deciders=<extracted>; run render-adr <bd-id>
      ELSE (bd record exists but no markdown — orphaned legacy):
        - DRY-RUN: report "would prompt operator for deciders on <bd-id>; would create .md"
        - APPLY: AskUserQuestion for deciders; bd update --set-metadata; run render-adr <bd-id> (creates the .md)

Final report:
  - Already migrated: N
  - Will-migrate (dry-run) / Newly migrated (apply): N
  - Will-recover / Newly recovered: N
  - Failed (needs manual attention): N + list bd-ids and reasons
```

Failure modes the report surfaces:

- bd `decision` bead with description missing required sections (Decision / Rationale / Alternatives Considered) — bd's `--validate` should have caught this, but pre-`--validate` legacy beads from older bd versions might trip.
- bd `decision` bead with no extractable Deciders in .md AND (under `--apply`) operator declines to provide → log as failed; continue.
- After migration, render-adr's output and the .md still don't match → drift detected; surface for manual review (likely operator-edited the .md after capture-adrs and we couldn't reconcile).

**Session-context constraint:** Under `--apply`, the orphan-bead recovery path calls `AskUserQuestion` to prompt the operator for missing deciders. This inherits `capture-adrs`'s INV-A19 constraint: `/adr migrate --apply` MUST run in the user's main session context. If `AskUserQuestion` is not in scope (Codex without main-session glue, subagent-dispatched migration), the orphan-recovery branch MUST fail fast with a clear error ("`/adr migrate --apply` requires main session context for operator prompts; run in primary session or use dry-run").

The dry-run path has no such constraint — it can run in any context (Codex, subagents, CI preview).

Run order per consumer repo:

1. Upgrade dev-flow plugin (PR landing this spec).
2. Run `/adr init` (per-repo bootstrap; copies formula).
3. Run `/adr migrate` (defaults to dry-run preview).
4. Review the preview; run `/adr migrate --apply` when comfortable.
5. Commit the resulting .md updates + bd state changes; push.

## Open questions / future work

- **Discussion threads on Proposed ADRs:** ADRs sometimes accumulate discussion before acceptance. bd notes can hold comments; should the renderer surface them as a `## Discussion` section for Proposed ADRs? Deferred — start with bare Proposed → Accepted, add discussion rendering if the team uses it.
- **ADR templates per category:** Some teams differentiate decision types (architectural / process / tooling). A single `formula-adr` covers all; per-category variants are future work if needed.
- **Linking ADRs to spec sections:** ADRs reference the spec via `--spec-id` (today's pattern). A "ADRs informed by this spec" reverse query is possible via `bd list --spec <spec-path>` but not currently materialized in the rendered markdown. Could be added to render-adr's References section if useful.

## Testing strategy

| Layer | Test |
|-------|------|
| Formula validity | `bd formula show formula-adr` succeeds; `bd mol pour formula-adr --var ... --dry-run` produces a valid `decision` bead spec with all variables substituted. |
| Render-adr | Render a synthetic test ADR; diff the output against an expected fixture. Verify each path: open / closed / superseded / deprecated; with/without addenda. |
| /adr commands | Manual: each subcommand against a synthetic test ADR. `--apply` semantics; idempotency of `/adr render`. |
| adr-doctor extensions | Synthetic ADRs that trip each new check; verify failure detection + non-failure on clean state. |
| Migration | `/adr migrate` (dry-run) on this repo's 5 ADRs reports expected actions; `/adr migrate --apply` produces expected `bd update` calls and re-renders. |
| Cross-platform | Codex consumers: `/adr` commands use bd + bash + jq + standard utilities; no Claude-Code-specific dependencies. Codex should work; verify manually. |
