# evolve-adr

Reference skill for the ADR lifecycle. bd is the source of truth for ADR content;
markdown files under `docs/adr/` are derived views regenerated by `render-adr`.

Full spec: `docs/superpowers/specs/2026-05-22-adr-evolution-design.md`

## Overview

ADRs in dev-flow are stored as bd `decision` beads. The bd record (title,
description, labels, metadata, notes, dep edges) is canonical. The markdown
file is a rendered view — never edit it by hand. Drift between the bd state
and the committed markdown is a lint failure caught by `adr-doctor`.

This pattern inverts the original `capture-adrs` approach (markdown-canonical,
drifting bd copy). Under the new model: write to bd, render to markdown, commit
both.

## bd as Source of Truth — What Lives Where

| ADR concept | bd-native representation |
|-------------|--------------------------|
| Title | bd bead title |
| Body (Context / Decision / Rationale / Alternatives / Consequences) | `--description` (markdown sections) |
| Status | Computed at render time from bd lifecycle + dep edges + labels |
| Date | `closed_at` if closed; `created_at` if still Proposed |
| Deciders | Metadata key `adr_deciders` (comma-separated names) |
| Supersession edges | `bd dep add <new> <old> --type=supersedes` |
| Addenda | `bd note <id> "addendum: <text>"` |
| Markdown file | `docs/adr/<bd-id>-<slug>.md` — derived view only |

The markdown header carries an `<!-- adr-render: source=bd:<id>; do not edit
manually; use /adr update <id> -->` marker on line 2 of every rendered file so
that readers and tooling know the file is generated.

## When to Use Each Entry Point

Three ways to create an ADR exist; pick based on timing and context:

### `capture-adrs` — retrospective, hook-triggered

Use after `writing-plans` finishes (the `nudge-adr-capture` hook fires a
reminder). The decision is already final — the brainstorming session concluded
and the spec records it. `capture-adrs` extracts candidates, walks you through
per-candidate approval via `AskUserQuestion`, creates each via `bd create
--type decision` (composing the body inline), closes the bead immediately
(auto-Accepted), then calls `render-adr`. Output: one or more Accepted ADRs
from a single spec pass.

Requires main session context (`AskUserQuestion` must be in scope; per INV-A19).

### `/adr propose` — forward-looking, decision under discussion

Use when a decision is being considered but not yet final. Creates the bd bead
and leaves it open (bd status=open → renders as `Status: Proposed`). The team
can discuss, update the draft via `/adr update <id>`, and later commit with
`/adr accept <id>`. Alternatively, if the team decides against it, reject via
the bd commands described under [Rejection](#rejection-path).

### `/adr new` — manual retrospective outside capture-adrs

Use when you want to document a past decision without going through the
`capture-adrs` scan-and-approve loop (e.g., documenting an implicit decision
discovered during a review, or creating an ADR from outside a writing-plans
session). Creates the bead, closes it immediately (Accepted), and renders.
Same end state as `capture-adrs` but driven manually.

## Status Mechanics

ADR status is **computed at render time** from bd-native signals. It is not
stored as a metadata field. The renderer applies this logic:

```text
if bd.status == "open":
    return "Proposed"
if bd has incoming supersedes edge:
    return "Superseded by <new-id>"
if bd.labels includes "adr:rejected":
    return "Rejected"
if bd.labels includes "adr:deprecated":
    return "Deprecated"
# closed, no special markers:
return "Accepted"
```

### Five terminal states

| Status | bd signals |
|--------|------------|
| Proposed | `bd.status == open` |
| Accepted | `bd.status == closed`, no `adr:rejected`/`adr:deprecated` label, no incoming supersedes edge |
| Superseded | `bd.status == closed` AND incoming `supersedes` dep edge exists |
| Rejected | `bd.status == closed` AND label `adr:rejected` |
| Deprecated | `bd.status == closed` AND label `adr:deprecated` |

Superseded, Rejected, and Deprecated are distinct terminal states with
different semantics: Superseded means a replacement ADR exists; Rejected means
the team decided against the proposal; Deprecated means an Accepted decision
has been retired without a direct replacement.

### Rejection path

No `/adr reject` command exists in v1. To reject a Proposed ADR:

```bash
bd update <id> --add-label adr:rejected
bd close <id> --reason="Rejected: <why>"
render-adr <id>
```

(`--add-label` is additive — preserves existing labels. Distinct from
`--set-labels` which replaces.)

## Supersession Semantics

Supersession links one ADR to its replacement via a bd dep edge.

To supersede an existing ADR with a new one:

```bash
# Use /adr supersede <old-id> (interactive) OR run manually:
bd dep add <new-id> <old-id> --type=supersedes
bd close <old-id> --reason="Superseded by <new-id>"
render-adr <old-id>
render-adr <new-id>
```

The renderer queries:

- Outgoing supersedes edges (from old to new) to emit `Supersedes: <old-id>` in
  the new ADR's References section.
- Incoming supersedes edges on the old ADR via
  `bd dep list <old-id> --direction=up --type=supersedes` to emit
  `Superseded by: <new-id>` in the old ADR's References section.

Both markdown files must be re-rendered after the dep edge is added. The
`/adr supersede <old-id>` command handles both renders automatically.

## Addenda

An addendum is an additive clarification to an Accepted ADR — it extends or
refines without replacing the decision. Use addenda when:

- A follow-up experiment confirms or sharpens a consequence.
- An implementation note corrects a minor inaccuracy in the Alternatives section.
- New context is relevant but the decision itself stands.

```bash
bd note <id> "addendum: <prose text here>"
render-adr <id>
```

The literal prefix `addendum:` followed by a space (lowercase) is required. The
renderer collects all notes with this prefix and surfaces them under
`## Addenda` in the markdown output.

**Addendum vs. supersession:** use an addendum for additive clarification that
leaves the decision intact. Use supersession when the structural decision
changes — a new approach replaces the old one, a different library is chosen,
or the architecture shifts materially.

## Migration

`/adr migrate` backfills legacy ADRs (created via the pre-formula
`capture-adrs` path) to the new envelope. It is idempotent and repo-portable.

```bash
/adr migrate            # dry-run preview — no mutations
/adr migrate --apply    # mutate bd metadata + re-render markdown
```

**What it does per legacy bead:**

1. Check `metadata.adr_deciders` — if set, bead is already migrated; skip.
2. Search `docs/adr/` for `<bd-id>-*.md`.
3. If file found: parse `**Deciders:**` line → `bd update <id>
   --set-metadata adr_deciders=<extracted>` → `render-adr <id>`.
4. If no file (orphaned legacy): dry-run reports the gap; `--apply` prompts
   for deciders via `AskUserQuestion`, then updates + renders.

**Session-context constraint (INV-A19):** the orphan-recovery branch under
`--apply` calls `AskUserQuestion`. Run `/adr migrate --apply` in the user's
main session context. If `AskUserQuestion` is not in scope (Codex subagent,
CI), the orphan-recovery branch fails fast:

```text
/adr migrate --apply requires main session context for operator prompts;
run in primary session or use dry-run.
```

The dry-run path has no such constraint and runs in any context.

**Run order for a consumer repo:**

1. Upgrade dev-flow plugin.
2. Run `/adr migrate` (dry-run). (`/adr init` is a no-op — `decision` is a
   built-in bd type and ADR bodies are composed inline; no formula to copy.)
3. Review preview; run `/adr migrate --apply` when satisfied.
4. Commit resulting `.md` updates + bd state; push.

## Edge Cases

### Codex compatibility

All `/adr` subcommands use `bd`, `bash`, `jq`, and standard POSIX utilities —
no Claude Code-specific APIs. `render-adr` is a plain bash script. These work
in Codex with no special glue. The single main-session dependency is
`AskUserQuestion` in:

- `capture-adrs` (approval loop)
- `/adr migrate --apply` (orphan-recovery prompts)

In Codex, skip those interactive paths (dry-run migration; use direct bd
commands for capture instead of the skill's approval loop).

### adr-doctor invariants

New checks added by this spec (in `dev-flow/scripts/adr-doctor.sh`):

| INV | Check |
|-----|-------|
| INV-A20 | `## Context` heading present in each decision bead's description |
| INV-A21 | `## Consequences` heading present in each decision bead's description |
| INV-A22 | Committed markdown matches `render-adr` output (drift detection) |
| INV-A23 | Bead with `adr:deprecated` label must have `bd.status == closed` |
| INV-A24 | `metadata.adr_deciders` non-empty for closed decision beads |

INV-A24 is warning-only for pre-migration beads; error after `/adr migrate
--apply` completes.

### Updating an existing ADR body

```bash
/adr update <id>    # interactive — prompts for which section + new prose
```

Internally: re-composes the full description body and pipes it via
`bd update <id> --body-file -` (shell-escape-safe for multi-line prose).
Then calls `render-adr <id>`. Commit both the bd state and the re-rendered
markdown.

### Re-rendering without mutation

```bash
/adr render <id>    # re-render only; no bd mutation
```

Use to fix INV-A22 drift detected by adr-doctor when the bd state is correct
but the committed markdown is stale.
